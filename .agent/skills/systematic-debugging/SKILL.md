# Systematic Debugging — 系统化调试

## 概览

随机修改代码只会浪费时间并引入新的 Bug。快速修补只会掩盖底层问题。

**核心原则：** 在尝试修复之前，**必须**找到根本原因。只修复症状就是失败。

**违反此流程的字面要求，即违反了调试的精神。**

> **来源**: 改编自 [Agent Superpowers - systematic-debugging](https://github.com/AgentSuperPowers/superpowers/tree/main/skills/systematic-debugging)
> **改编说明**: 原版面向 Claude Code，本版改为跨平台通用格式，适用于 所有 AI 编程助手。

## The Iron Law (铁律)

```
NO FIXES WITHOUT ROOT CAUSE INVESTIGATION FIRST
(没有查明根本原因之前，绝不进行修复)
```

如果你还没有完成阶段 1，你就不能提出任何修复方案。

## 何时激活

用于**任何**技术问题：
- 测试失败
- 生产环境的 Bug
- 意外行为
- 性能问题
- 构建失败
- 集成问题

**在以下情况尤其需要使用：**
- 时间紧迫（紧急情况容易让人靠猜来修复）
- "只需一个快速修复" 看起来很明显时
- 你已经尝试了多次修复但都无效时
- 上一次修复不起作用时
- 你不完全理解这个问题时

**以下情况也不要跳过：**
- 问题看起来很简单（简单的 Bug 也有根本原因）
- 你很匆忙（越急越容易出错返工）
- 经理希望现在就修好（系统化调试比瞎碰要快得多）

## The Four Phases (四个阶段)

你**必须**在进行下一个阶段之前完成当前阶段。

### 阶段 1: Root Cause Investigation (根本原因调查)

**在尝试任何修复之前：**

1. **仔细阅读错误信息**
   - 不要跳过错误或警告信息
   - 它们通常包含确切的解决方案
   - 完整阅读堆栈跟踪
   - 注意行号、文件路径、错误代码

2. **稳定复现**
   - 你能稳定触发它吗？
   - 确切的步骤是什么？
   - 每次都会发生吗？
   - 如果无法复现 → 收集更多数据，不要靠猜

3. **检查最近的变更**
   - 有什么变动可能导致了这个问题？
   - Git diff，最近的提交
   - 新的依赖项、配置变更
   - 环境差异

4. **在多组件系统中收集证据**

   **当系统有多个组件时（如 CI → 构建 → 签名，API → 服务 → 数据库）：**

   **在提出修复方案之前，添加诊断性日志：**
   ```text
   对于每个组件边界：
     - 记录什么数据进入了组件
     - 记录什么数据离开了组件
     - 验证环境/配置的传递
     - 检查每一层的状态

   运行一次以收集证据，查明到底是在哪里断开的
   然后分析证据以确定失败的组件
   然后调查那个特定的组件
   ```

5. **追踪数据流**

   **当错误在调用栈深处时：**
   - 错误值是从哪里产生的？
   - 是什么代码用这个错误值调用了当前函数？
   - 一直向上追溯，直到找到源头
   - 在源头修复，而不是在症状处修复

### 阶段 2: Pattern Analysis (模式分析)

**在修复之前找到模式：**

1. **寻找正常工作的例子**
   - 在同一代码库中找到相似且正常工作的代码
   - 什么是有效的且与损坏的部分相似的？

2. **与参考实现对比**
   - 如果正在实现某种模式，请**完整地**阅读参考实现
   - 不要跳读 - 读懂每一行
   - 在应用之前充分理解该模式

3. **识别差异**
   - 正常工作和出错的部分有什么区别？
   - 列出每一个区别，不管多小
   - 不要假设 "那不可能有影响"

4. **理解依赖关系**
   - 这还需要哪些其他组件？
   - 需要什么设置、配置、环境？
   - 它做了什么假设？

### 阶段 3: Hypothesis and Testing (假设与测试)

**科学方法：**

1. **形成单一假设**
   - 清晰地陈述："我认为 X 是根本原因，因为 Y"
   - 写下来
   - 要具体，不要含糊

2. **最小化测试**
   - 做出**最小**的改变来测试假设
   - 一次只改变一个变量
   - 不要一次修复多处

3. **验证后继续**
   - 有效吗？是 → 进入阶段 4
   - 无效？形成**新的**假设
   - **不要**在上面叠加更多的修复尝试

4. **当你不懂时**
   - 说 "我不理解 X"
   - 不要假装懂
   - 请求帮助或提供更多上下文
   - 进行更多研究

### 阶段 4: Implementation (实现)

**修复根本原因，而不是症状：**

1. **创建失败的测试用例**
   - 最简单的复现步骤
   - 尽可能使用自动化测试
   - 如果没有框架，使用一次性测试脚本
   - 必须在修复之前拥有测试
   - 使用 `test-driven-development` 技能编写正确的失败测试

2. **实现单一修复**
   - 解决已确定的根本原因
   - 一次只做一个改变
   - 没有 "既然我在这儿顺便改一下" 的改进
   - 不捆绑重构

3. **验证修复**
   - 测试现在通过了吗？
   - 其他测试没有被破坏？
   - 问题真的解决了吗？

4. **如果修复不起作用**
   - **停下**
   - 统计：你已经尝试了多少次修复？
   - 如果 < 3 次：回到阶段 1，利用新信息重新分析
   - **如果 ≥ 3 次：停下来质疑架构（见下方的步骤 5）**
   - 在没有进行架构讨论之前，不要尝试第 4 次修复

5. **如果 3 次以上修复都失败了：质疑架构**

   **表明存在架构问题的模式：**
   - 每次修复都在不同地方暴露出新的共享状态/耦合/问题
   - 修复需要 "大规模重构" 才能实现
   - 每次修复都会在其他地方产生新的症状

   **停下来质疑基础设计：**
   - 这种模式从根本上说合理吗？
   - 我们是不是仅仅因为 "惯性" 而坚持使用它？
   - 我们是应该重构架构，还是继续修复症状？

   这**不是**假设失败 - 这是架构错误。

## 红旗信号 — 停下来并遵循流程

如果你发现自己有以下想法：
- "先快速修好，以后再调查"
- "试着改一下 X 看看行不行"
- "加好几个修改，跑一下测试"
- "跳过测试，我会手动验证"
- "大概是 X 吧，让我修一下"
- "我不完全懂但这个可能会管用"
- "这里有几个主要问题：[在未调查前列出修复方案]"
- 在追踪数据流之前就提出解决方案
- **"再试最后一次修复" (当已经尝试了 2 次以上时)**
- **每次修复都在不同的地方揭露出新的问题**

**所有这些都意味着：停下。回到阶段 1。**

**如果 3 次以上的修复失败：** 质疑架构 (见阶段 4 步骤 5)

## 常见借口 vs 现实

| 借口 | 现实 |
|------|------|
| "问题很简单，不需要流程" | 简单问题也有根本原因。对于简单的 Bug，流程执行得很快。 |
| "情况紧急，没时间走流程" | 系统化调试比猜猜看的盲目尝试要快得多。 |
| "先试试这个，不行再调查" | 第一次的修复尝试往往会带偏方向。一开始就做对。 |
| "确认修复有效后我再补测试" | 未经测试的修复无法持久。先测试能证明修复真的有效。 |
| "一次改多个地方省时间" | 无法分离出到底是哪个修改起作用了。而且会引发新的 Bug。 |
| "参考代码太长，我自己改改模式" | 一知半解必然导致 Bug。完整阅读参考实现。 |
| "我看出问题了，让我直接修" | 看到症状 ≠ 理解根本原因。 |
| "再试一次修复" (失败 >2 次后) | 3次以上的失败 = 架构问题。质疑模式，而不是继续修。 |

## 支持性技术

这些技术是系统化调试的一部分（原版包含详细文档）：
- **Root Cause Tracing (根本原因追踪)**: 反向追踪调用栈寻找原始触发点。
- **Defense in Depth (纵深防御)**: 在找到根本原因后，在多个层级添加验证。
- **Condition-based Waiting (条件轮询)**: 用条件轮询代替随意的超时等待。
