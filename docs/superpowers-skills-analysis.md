# Superpowers Skills 调研分析报告

> 项目地址：https://github.com/obra/superpowers
> 调研时间：2026-02-22

---

## 一、项目概览

**Superpowers** 是一个为 AI 编码代理（如 Claude Code、Cursor、Codex 等）设计的 **完整软件开发工作流框架**。它通过一组可组合的 "skills"（技能）和初始指令，让 AI 代理按照严格的方法论进行软件开发，而不是一上来就胡乱写代码。

### 核心理念

| 理念 | 说明 |
|------|------|
| **测试驱动开发 (TDD)** | 先写测试，永远如此 |
| **系统化优于随意** | 过程胜于猜测 |
| **简化复杂度** | 简洁是首要目标 |
| **证据优于断言** | 在宣称成功之前先验证 |

### 工作方式

1. AI 代理启动后，**不会直接跳入写代码**，而是先退后一步，问你真正想做什么
2. 通过对话梳理出一份 **设计规格**，并分段呈现给你审阅
3. 你签字认可后，代理编写一份 **实施计划**（足够清晰到一个"缺乏经验的初级工程师"都能跟着做）
4. 然后启动 **子代理驱动开发** 流程，让代理分任务执行、检查和审查

## 三、各 Skill 详细分析

### 1. 🎯 `using-superpowers` — 系统入口 & 调度中心

**触发时机：** 每次对话开始时

**核心功能：**
- 这是整个 skills 系统的 **元调度器**，负责决定何时调用哪个 skill
- 使用 **极其强硬的语气** 要求代理必须检查并使用相关 skills：
  > *"如果你认为有哪怕 1% 的可能性某个 skill 适用，你**绝对必须**调用它。这不可商量，不可选择。"*

**关键规则：**
- 在任何响应或操作之前，先检查并调用相关 skills
- 优先级：流程技能（brainstorming, debugging）> 实现技能
- 分为 **刚性技能**（TDD、调试，必须严格遵循）和 **柔性技能**（模式类，可灵活适配）

**反合理化表：**
| 想法 | 现实 |
|------|------|
| "这只是个简单问题" | 问题也是任务，先检查 skills |
| "我需要先了解更多上下文" | 检查 skills **在**提出澄清问题**之前** |
| "这不需要正式的 skill" | 如果 skill 存在，就用它 |
| "Skill 过于大材小用" | 简单的事情会变复杂 |

---

### 2. 💡 `brainstorming` — 创意到设计的转化器

**触发时机：** 编写代码之前，有新想法或需求时激活

**核心功能：**
- 通过苏格拉底式对话，将 **粗略的想法** 逐步精炼为 **完整的设计文档**
- 设有 **硬门禁**：在设计获批之前，**绝对不能** 写任何代码、搭建项目

**流程：**
```
探索项目上下文 → 逐个提问澄清 → 提出 2-3 种方案及权衡 
→ 分段呈现设计 → 用户逐段审批 → 写入设计文档 → 调用 writing-plans
```

**关键原则：**
- **一次一个问题** — 不要同时抛出多个问题压倒用户
- **优先多选题** — 比开放式问题更容易回答
- **YAGNI 原则** — 无情删除不必要的功能
- 设计文档保存到 `docs/plans/YYYY-MM-DD-<topic>-design.md`

**反模式警惕：** "这太简单了，不需要设计" → **每个项目**都必须走这个流程，即使是最简单的 todo 应用

---

### 3. 🌳 `using-git-worktrees` — 隔离工作区管理

**触发时机：** 设计获批后，在实施之前

**核心功能：**
- 使用 Git Worktree 创建 **隔离的工作空间**，在新分支上工作
- 共享同一仓库，无需反复 checkout 分支

**流程：**
```
检测项目名 → 创建 worktree 和分支 → 运行项目 setup（npm install 等）
→ 验证测试基线 → 报告位置
```

**安全验证：**
- 检查 worktree 目录是否已被 .gitignore 忽略
- 确保测试在开始前全部通过（干净基线）
- 如果测试失败，先报告再决定是否继续

---

### 4. 📋 `writing-plans` — 实施计划编写器

**触发时机：** 有了设计规格或需求之后，在写代码之前

**核心功能：**
- 编写 **极其详细** 的实施计划，假设执行者 **完全没有项目上下文**
- 每个步骤都是 **2-5 分钟的小任务**

**任务粒度示例：**
```
步骤 1: 写失败的测试          ← 一个步骤
步骤 2: 运行测试确认失败      ← 一个步骤  
步骤 3: 写最少代码让测试通过  ← 一个步骤
步骤 4: 运行测试确认通过      ← 一个步骤
步骤 5: 提交                  ← 一个步骤
```

**每个任务包含：**
- 精确的文件路径
- 完整的代码（不是"添加验证逻辑"这种模糊描述）
- 可执行的命令和预期输出
- 设计原则：DRY、YAGNI、TDD、频繁提交

**计划完成后提供两种执行选择：**
1. **子代理驱动** — 在当前会话中，每个任务调度新子代理
2. **并行会话** — 在新会话中使用 `executing-plans` 分批执行

---

### 5. 🤖 `subagent-driven-development` — 子代理驱动开发（推荐方式）

**触发时机：** 计划编写完成后，用户选择在当前会话执行

**核心功能：**
- 每个任务调度 **全新的子代理**（避免上下文污染）
- 实施后进行 **双阶段审查**：先规格合规性审查，再代码质量审查

**流程：**
```
读取计划 → 提取所有任务（含完整文本）→ 创建 TodoWrite

对于每个任务：
  ├─ 调度实现子代理（implementer-prompt.md）
  ├─ 子代理有问题？ → 回答并提供上下文
  ├─ 子代理实现、测试、提交、自审
  ├─ 调度规格审查子代理（spec-reviewer-prompt.md）
  │   └─ 不合规？ → 修复 → 重新审查
  ├─ 调度代码质量审查子代理（code-quality-reviewer-prompt.md）
  │   └─ 不通过？ → 修复 → 重新审查
  └─ 标记任务完成

所有任务完成 → 最终代码审查 → finishing-a-development-branch
```

**三个 Prompt 模板：**
- `implementer-prompt.md` — 实现者子代理
- `spec-reviewer-prompt.md` — 规格合规审查子代理
- `code-quality-reviewer-prompt.md` — 代码质量审查子代理

**优势：**
- 每个子代理有 **全新的上下文**，不会混乱
- 控制器提前提取所有任务的完整文本，不让子代理自己去读文件
- 双阶段审查防止过度构建或构建不足

**关键红线：**
- ❌ 绝不在 main/master 分支上直接开始实现
- ❌ 绝不跳过任何审查（规格合规 OR 代码质量）
- ❌ 绝不在规格合规通过之前就开始代码质量审查（顺序不能错）
- ❌ 绝不并行调度多个实现子代理（会冲突）

---

### 6. 📦 `executing-plans` — 分批计划执行器

**触发时机：** 用户选择在新会话中执行计划

**核心功能：**
- 加载计划 → 批判性审查 → 分批执行 → 报告并等待反馈

**与 subagent-driven-development 的区别：**

| 特性 | `subagent-driven-development` | `executing-plans` |
|------|-------------------------------|-------------------|
| 会话 | 同一会话 | 新的独立会话 |
| 执行方式 | 每任务调度新子代理 | 直接分批执行（默认 3 个） |
| 审查方式 | 双阶段子代理自动审查 | 每批完成后等待人工反馈 |
| 迭代速度 | 更快（无需人工介入） | 需要人工检查点 |

**关键原则：** 遇到阻塞就 **立刻停下来问**，绝不猜测

---

### 7. 🧪 `test-driven-development` — TDD 铁律

**触发时机：** 任何实现过程中

**核心功能：** 强制执行严格的 RED-GREEN-REFACTOR 循环

**铁律：** `先写失败的测试，再写生产代码。没有例外。`

**RED-GREEN-REFACTOR 周期：**
```
🔴 RED:    写一个最小的失败测试 → 运行确认它失败
🟢 GREEN:  写最少代码让测试通过 → 运行确认通过
🔵 REFACTOR: 清理代码（去重、改名、提取） → 保持测试绿色
```

**铁腕规则：**
- 在测试之前写了代码？**删掉它**。重新开始。
- 不保留它作为"参考"
- 不"改编"它
- 不看它
- 删除就是删除

**好测试的标准：**
- 测试一个行为
- 清晰的名字
- 使用真实代码（除非不可避免，不用 mock）

---

### 8. 🐛 `systematic-debugging` — 系统化调试

**触发时机：** 遇到任何技术问题时

**核心功能：** 4 阶段根因分析流程，禁止在找到根因前就尝试修复

**铁律：** `没有根因调查，就不能提出修复方案`

**四阶段流程：**

```
阶段 1: 根因调查
  ├─ 仔细阅读错误信息（不要跳过！）
  ├─ 稳定复现
  ├─ 检查最近的变更（git diff）
  ├─ 多组件系统：在每个组件边界添加诊断日志
  └─ 追踪数据流（从错误往上追溯到源头）

阶段 2: 模式分析
  ├─ 找到同代码库中的正常工作的类似代码
  ├─ 对比参考实现（完整阅读，不要跳读！）
  └─ 找出所有差异

阶段 3: 假设与测试
  ├─ 形成单一假设："我认为 X 是根因，因为 Y"
  ├─ 做最小的改动来测试假设
  └─ 一次只改一个变量

阶段 4: 实施修复
  └─ 确认根因后才开始修复
```

**特别适用于：**
- 时间压力下（紧急情况让猜测变得诱人）
- "就快速修一下"似乎显而易见时
- 已经试了多种修复都没成功时

---

### 9. ✅ `verification-before-completion` — 验证核查门

**触发时机：** 即将声称工作完成、已修复、已通过时

**核心功能：** 在任何成功声明之前，强制要求运行验证命令并确认输出

**铁律：** `没有新鲜的验证证据，就不能声称完成`

**门控函数：**
```
1. 确定: 什么命令能证明这个说法？
2. 执行: 运行完整命令（新鲜的、完整的）
3. 阅读: 完整输出，检查退出码，计数失败
4. 校验: 输出是否确认了声明？
5. 然后: 才能做出声明
```

**常见的虚假完成模式：**
| 声明 | 需要 | 不够 |
|------|------|------|
| 测试通过 | 测试命令输出：0 失败 | 以前的运行、"应该能过" |
| Bug 已修复 | 测试原始症状：通过 | 代码改了，假设修好了 |
| 构建成功 | 构建命令：exit 0 | Linter 通过 ≠ 构建成功 |

**红线词汇：** 使用 "应该"、"可能"、"似乎" 都是危险信号

> 来源：24 条失败记忆中总结。人类说过"我不相信你"——信任一旦打破就很难修复。

### 10. 🔍 `requesting-code-review` — 发起代码审查

**触发时机：**
- **强制：** 每个任务完成后、重大功能完成后、合并到 main 之前
- **可选：** 卡住时（新视角）、重构前、修复复杂 bug 后

**核心功能：** 通过获取 git SHA、调度 code-reviewer 子代理来进行代码审查

**反馈处理：**
- 🔴 **Critical** → 立即修复
- 🟠 **Important** → 继续前修复
- 🟡 **Minor** → 记录稍后处理
- 反驳？可以，但需要技术理由

---

### 11. 📝 `receiving-code-review` — 接收代码审查

**触发时机：** 收到代码审查反馈时

**核心功能：** 以技术评估（而非社交表演）的方式处理反馈

**响应模式：**
```
1. 阅读: 完整反馈，不要反应
2. 理解: 用自己的话重述需求（或提问）
3. 验证: 对照代码库现实检查
4. 评估: 对当前代码库来说技术上合理吗？
5. 响应: 技术确认或有理据的反驳
6. 实施: 一次一个，每个都测试
```

**禁止的回应：**
- ❌ "你说得太对了！" — 表演性
- ❌ "好观点！" / "很好的反馈！" — 表演性
- ❌ "让我现在就实施" — 在验证之前
- ✅ 直接重述技术需求，开始干活

**YAGNI 检查：** 审查者建议的"专业"功能是否真的需要？

---

### 12. 🔀 `dispatching-parallel-agents` — 并行代理调度

**触发时机：** 多个独立问题需要同时解决时

**核心功能：** 将不相关的问题分配给不同子代理并发执行

**使用条件：**
- ✅ 3+ 个测试文件因不同根因失败
- ✅ 多个子系统独立故障
- ✅ 每个问题可以独立理解和解决
- ❌ 故障相关联（修一个可能修全部）
- ❌ 代理之间会互相干扰（编辑相同文件）

**代理 Prompt 要求：**
1. **聚焦** — 一个清晰的问题域
2. **自包含** — 包含理解问题所需的所有上下文
3. **明确输出** — 代理应该返回什么？

**实际案例：** 6 个测试失败 → 分 3 组并行调度 → 全部独立修复 → 零冲突 → 3 倍速度

---

### 13. 🏁 `finishing-a-development-branch` — 完成开发分支

**触发时机：** 所有任务完成、测试通过后

**核心功能：** 引导开发工作的收尾，提供结构化选项

**流程：**
```
验证测试 → 确定基础分支 → 呈现 4 个选项 → 执行选择 → 清理 worktree
```

**固定 4 个选项：**
| 选项 | 合并 | 推送 | 保留 Worktree | 清理分支 |
|------|------|------|--------------|---------|
| 1. 本地合并 | ✓ | - | - | ✓ |
| 2. 创建 PR | - | ✓ | ✓ | - |
| 3. 保持现状 | - | - | ✓ | - |
| 4. 丢弃工作 | - | - | - | ✓ (强制) |

**选项 4 安全措施：** 需要用户输入 "discard" 确认，防止误删

---

### 14. 🛠️ `writing-skills` — Skill 编写指南

**触发时机：** 需要创建新的 skill 时

**核心功能：** 将 TDD 方法论应用于 **流程文档编写**

**TDD 映射：**
| TDD 概念 | Skill 创建 |
|----------|-----------|
| 测试用例 | 压力场景 + 子代理 |
| 生产代码 | Skill 文档 (SKILL.md) |
| RED（失败） | 代理在没有 skill 时违规（基线） |
| GREEN（通过） | 代理有 skill 时遵从 |
| REFACTOR | 堵住漏洞但保持合规 |

**创建 Skill 的条件：**
- ✅ 技术不是直觉上显而易见的
- ✅ 会在多个项目中被引用
- ✅ 模式适用面广
- ❌ 一次性方案
- ❌ 项目特定的约定（放 CLAUDE.md 里）

## 四、Skills 之间的协作关系

### 主工作流管线（The Main Pipeline）

整个 Superpowers 的核心是一条 **严格有序的开发管线**，每个 skill 在管线的特定位置被激活：

```
┌─────────────────────────────────────────────────────────────────────┐
│                        完整开发生命周期                              │
│                                                                     │
│  ①using-superpowers ─── 元调度器，在每步之前检查该用哪个 skill       │
│         │                                                           │
│         ▼                                                           │
│  ②brainstorming ─────── 将想法精炼为设计文档                        │
│         │                                                           │
│         ▼                                                           │
│  ③using-git-worktrees ─ 创建隔离工作区                              │
│         │                                                           │
│         ▼                                                           │
│  ④writing-plans ──────── 编写详细实施计划                           │
│         │                                                           │
│         ├──────────────────┬─────────────────────┐                  │
│         ▼                  ▼                     │                  │
│  ⑤subagent-driven    ⑥executing-plans           │                  │
│    -development         (分批+人工检查)           │                  │
│    (子代理+自动审查)                              │                  │
│         │                  │                     │                  │
│         ▼                  ▼                     │                  │
│  ⑦finishing-a-development-branch ◄───────────────┘                  │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 横向贯穿的支撑技能

除了主管线之外，有几个 skill 是 **横向贯穿** 整个流程的，在任何阶段都可能被触发：

```
                    主管线各阶段
                ┌──────────────────┐
                │ brainstorming    │
                │ writing-plans    │
                │ subagent-dev     │
                │ executing-plans  │
                │ finishing-branch │
                └──────────────────┘
                        │
        ┌───────────────┼───────────────┐
        ▼               ▼               ▼
┌──────────────┐ ┌─────────────┐ ┌───────────────────┐
│     TDD      │ │  系统化调试  │ │ 验证核查门         │
│ (每次实现时) │ │(遇到问题时) │ │(每次声称完成时)    │
└──────────────┘ └─────────────┘ └───────────────────┘
```

| 横向 Skill | 嵌入位置 |
|-----------|---------|
| `test-driven-development` | 在 subagent-dev 和 executing-plans 的每个实现步骤中 |
| `systematic-debugging` | 在任何阶段遇到问题时触发 |
| `verification-before-completion` | 在每次声称"完成"之前触发 |
| `requesting-code-review` | 在 subagent-dev 的每个任务后和合并前 |
| `receiving-code-review` | 收到审查反馈时 |
| `dispatching-parallel-agents` | 在遇到多个独立问题时（主要在调试和实现阶段） |

### Skill 之间的显式调用关系

每个 skill 在文档中都明确声明它会调用（`REQUIRED SUB-SKILL`）哪些其他 skill：

```
brainstorming
  └─ CALLS → writing-plans
                └─ CALLS → subagent-driven-development  (选项 1)
                │            ├─ USES → test-driven-development (每个任务)
                │            ├─ USES → requesting-code-review (每个任务后)
                │            └─ CALLS → finishing-a-development-branch
                │                         └─ CLEANS UP → using-git-worktrees
                │
                └─ CALLS → executing-plans  (选项 2)
                             ├─ USES → test-driven-development (每个任务)
                             └─ CALLS → finishing-a-development-branch
                                          └─ CLEANS UP → using-git-worktrees

using-superpowers
  └─ DISPATCHES → 所有其他 skill (根据上下文判断)

systematic-debugging
  └─ 独立触发，不调用其他 skill

verification-before-completion
  └─ 独立触发，不调用其他 skill

writing-skills
  └─ REQUIRES → test-driven-development (TDD 方法论的基础)
```

### 子代理驱动开发中的内部协作（最复杂的交互）

`subagent-driven-development` 是协作最密集的 skill，它在单个任务的执行中编排了 **3 种角色的子代理** 和 **多个 skill 的交叉使用**：

```
┌────────────────── 控制器（主代理）──────────────────┐
│                                                     │
│  对于每个 Task:                                     │
│                                                     │
│  ┌─ 实现者子代理 ─────────────────────────────────┐ │
│  │  • 遵循 TDD skill（写测试 → 失败 → 实现 → 通过）│ │
│  │  • 自我审查                                     │ │
│  │  • 提交代码                                     │ │
│  └─────────────────────────────────────────────────┘ │
│                      │                               │
│                      ▼                               │
│  ┌─ 规格审查子代理 ───────────────────────────────┐ │

│  │  • 不合规 → 实现者修复 → 重新审查               │ │
│  └─────────────────────────────────────────────────┘ │
│                      │                               │
│                      ▼                               │
│  ┌─ 代码质量审查子代理 ──────────────────────────┐  │

│  ← 贯穿整个过程 →                                   │

│                                                     │
└─────────────────────────────────────────────────────┘
```

## 五、设计哲学与执行机制分析

### 1. "反合理化"设计模式

Superpowers 最突出的设计特征是对 AI "合理化跳过规则"行为的 **预判和封堵**。几乎每个 skill 都包含：

| 组件 | 作用 |
|------|------|
| **Iron Law（铁律）** | 不可违反的核心规则，用代码块高亮 |
| **Rationalization Table（合理化表）** | 列出 AI 可能用来逃避规则的借口，逐一反驳 |
| **Red Flags（红线列表）** | 列出违规行为的预警信号 |
| **Common Mistakes（常见错误）** | 经验中积累的反面教材 |

**示例：** `verification-before-completion` 中：
> - 借口："应该能行了" → 现实：运行验证命令
> - 借口："我很有信心" → 现实：信心 ≠ 证据
> - 借口："就这一次" → 现实：没有例外

### 2. 三层质量保障

```
第 1 层: TDD（开发时）
  └─ 先写测试，确保每行代码都有测试覆盖

第 2 层: 双阶段代码审查（开发后）
  ├─ 规格合规审查：做的对不对？
  └─ 代码质量审查：做的好不好？

第 3 层: 验证核查门（每次声明前）
  └─ 用实际命令输出证明，而不是用语言声明
```

### 3. 上下文隔离策略

Superpowers 深刻理解 AI 代理的 **上下文窗口限制** 和 **上下文污染** 问题：

- **Git Worktree**：物理隔离工作目录，避免主分支被污染
- **Fresh Subagent**：每个任务都调度新的子代理，避免前一个任务的残留信息影响后续任务
- **控制器提供完整文本**：不让子代理自己去读计划文件，由控制器提前提取并提供，减少文件读取开销
- **Prompt 模板**：标准化的子代理指令格式，确保每次调度的一致性

### 4. 人类在环（Human-in-the-Loop）

虽然 Superpowers 追求 AI 的高度自治，但在关键节点设置了 **明确的人工检查点**：

| 检查点 | 位置 |
|--------|------|
| 设计审批 | brainstorming → 用户逐段审批设计 |
| 计划审查 | writing-plans → 用户确认计划后才开始执行 |
| 执行选择 | writing-plans → 用户选择 subagent-dev 或 executing-plans |
| 批间反馈 | executing-plans → 每 3 个任务暂停等待反馈 |
| 完成决策 | finishing-branch → 用户选择合并/PR/保留/丢弃 |
| 丢弃确认 | finishing-branch → 需输入 "discard" 确认 |

### 5. YAGNI 和 DRY 的激进执行

整个框架反复强调两个原则：
- **YAGNI (You Aren't Gonna Need It)**：不要提前构建不需要的功能
- **DRY (Don't Repeat Yourself)**：不重复自己

这在 `receiving-code-review` 中特别体现：审查者建议的"专业"功能，需要经过 YAGNI 检查——如果不是当前明确需要的，就不要加。

## 七、关键洞察与启示

### 对 AI 代理规则持久化的启示

结合你之前在 `ai-rules-persistent-guide.md` 中研究的 AI 规则注入策略，Superpowers 项目验证了以下关键发现：

#### 1. **规则必须是"不可商量的"，而不是"建议"**

普通的 system prompt 写法：
> "请尽量遵循 TDD 原则"

Superpowers 的写法：
> `NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST` — **违反这条规则的条文就是违反它的精神**

**启示：** 使用 `<EXTREMELY-IMPORTANT>` 标签、铁律代码块、全大写声明来标记不可商量的规则。

#### 2. **预判并封堵合理化路径**

AI 会在长对话中找到"合理的借口"来跳过规则。Superpowers 通过 **合理化防御表** 预先列出所有可能的借口并逐一反驳。

**启示：** 任何重要规则都需要附带一个 **"你可能会这么想，但..."** 的反驳表。

#### 3. **Skill 是 Context-on-Demand 模式**

Skills 不是一次性全部加载到上下文窗口中的。它们是 **按需加载** 的——只有当 `using-superpowers` 判断某个 skill 可能适用时才调用。

**启示：** 这是解决上下文窗口限制的优雅方案。将规则拆分为多个独立的 skill 文件，按需加载。

#### 4. **"过程即产品"**

Superpowers 的核心创新不是任何单一的技术技巧，而是将 **人类的软件工程最佳实践**（TDD、设计审查、代码审查、分支管理）编码为 AI 可以遵循的 **机械化流程**。每个 skill 都是一个"标准操作程序"（SOP）。

#### 5. **三种角色分离**

在 subagent-driven-development 中体现的 **实现者 / 规格审查者 / 代码质量审查者** 三角色分离，本质上是在 AI 系统中模拟了软件团队的 **角色制衡**。一个 AI "写代码"，另一个 AI "挑毛病"，第三个 AI "审美"。

## 九、不足与改进空间（批判性分析）

### 🔴 严重问题

#### 1. Token / 成本爆炸

这是 Superpowers 最大的隐性代价。`subagent-driven-development` 中，**每个任务** 需要调度至少 3 个子代理：

```
每个任务的成本 = 实现者子代理 + 规格审查子代理 + 代码质量审查子代理
                + 如果审查不通过 → 修复 + 重新审查（循环）

一个 10 任务的计划 = 最少 30 次子代理调用
                   + 最终整体审查
                   + 控制器自身的上下文消耗
```

**问题：** 对于中等复杂度的项目，一次完整的 Superpowers 流程可能消耗 **数十万到上百万 token**，折合 API 费用可能在几十美元级别。文档中完全没有讨论成本控制策略。

**改进建议：**
- 增加"轻量模式"：简单任务可以跳过审查或合并审查阶段
- 提供成本估算：在制定计划时预估 token 消耗
- 支持审查阈值：仅对复杂度超过阈值的任务启用双阶段审查

#### 2. 平台强绑定 (Platform Lock-in)

整个框架深度依赖 Claude Code 的特定功能：

| 依赖项 | Claude Code 特有 |
|--------|-----------------|
| `Skill` tool | 用于加载 skill 文件 |
| `TodoWrite` | 用于任务跟踪 |
| 子代理调度 (Task tool) | 用于 subagent-driven-dev |
| `CLAUDE.md` | 项目级配置 |

**问题：** 虽然 README 声称支持 Cursor、Codex、OpenCode，但核心工作流（特别是子代理驱动开发）高度依赖 Claude Code 的子代理机制。迁移到其他平台需要大量适配。

**改进建议：**
- 抽象出"代理能力接口"，而不是直接引用平台特定工具名
- 为每个 skill 提供"降级方案"——当平台不支持子代理时该怎么做

### 🟠 中度问题

#### 3. 小任务的流程过载 (Process Overhead)

`brainstorming` 明确要求 **"每个项目都必须走完流程，即使是最简单的 todo 应用"**。

考虑这个场景：

```
用户: "帮我把这个变量名从 foo 改成 bar"

Superpowers 要求的流程:
  1. brainstorming → 设计讨论变量重命名
  2. writing-plans → 编写重命名计划
  3. using-git-worktrees → 为变量重命名创建 worktree
  4. subagent-driven-dev → 调度子代理重命名
  5. 双阶段审查 → 审查重命名
  6. finishing-branch → 合并重命名

实际需要: 全局替换，2 秒搞定
```

**问题：** 缺乏 **复杂度分级机制**。一行代码的修改和重写整个模块走相同的流程，这本身就违反了 YAGNI 原则。

**改进建议：**
- 引入复杂度评估阶段（trivial / small / medium / large）
- trivial 和 small 变更可以跳过 brainstorming 和 worktree
- 让 `using-superpowers` 根据复杂度决定跳过哪些步骤

#### 4. TDD 教条主义

TDD skill 的铁律是 `NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST`，并且 **"没有例外"**。

但现实中很多场景并不适合严格 TDD：

| 场景 | TDD 困难程度 |
|------|------------|
| UI/前端交互 | 难以用单元测试覆盖视觉效果 |
| 数据管线/ETL | 依赖外部数据源，mock 成本高 |
| ML 训练脚本 | 输出不确定，难以写断言 |
| 配置文件修改 | 不需要测试 |
| 一次性脚本/原型 | 测试的 ROI 极低 |
| 基础设施代码 (IaC) | 测试需要真实环境 |

**问题：** 虽然 TDD skill 列出了 "例外（问你的人类伙伴）：一次性原型、生成代码、配置文件"，但这个例外列表太短了。框架没有提供在非 TDD 场景下的替代质量保障方案。

**改进建议：**
- 扩展例外列表，增加更多现实场景
- 提供 TDD 的"替代策略"：如手动验证检查清单、截图对比、smoke test
- 不要将"跳过 TDD = 偷懒"等价起来

#### 5. 假设干净的代码库起点

`using-git-worktrees` 要求 **"验证干净基线——运行测试确保 worktree 开始时全部通过"**。

**问题：** 现实中大量项目：
- 根本没有测试
- 测试是坏的 / flaky 的
- CI 一直红着
- 是遗留代码库，加测试本身就是一个大项目

当基线就是"红色"的时候，整个 Superpowers 流程会卡在第一步。文档只说了 "如果测试失败，报告并询问是否继续"，但没有提供在遗留代码库中如何有效工作的策略。

**改进建议：**
- 增加 `legacy-codebase-strategy` skill：如何在无测试代码库中渐进式引入测试
- 支持"局部基线"：只验证与当前变更相关的测试子集
- 提供 "golden file" 或 "characterization test" 等遗留代码测试模式

### 🟡 次要但值得关注的问题

#### 6. 反合理化可能适得其反

Superpowers 的反合理化设计是其最大创新之一，但存在矛盾：

```
设计意图:
  "不要跳过 TDD" → 列出所有借口并反驳

实际可能发生:
  AI 读到 "如果你想跳过，那就是合理化" 
  → AI 遇到真正不适合 TDD 的场景
  → AI 知道不应该跳过（因为规则说了不能合理化）
  → AI 强行写了一个没有意义的测试
  → 浪费时间 + 产生误导性的测试
```

**问题：** 当"遵守规则"本身变成了教条，AI 就失去了在 **真正例外场景** 中做出正确判断的能力。反合理化防御把"所有跳过规则的想法"都等同于"偷懒"，但有时跳过确实是正确的。

**改进建议：**
- 区分"纪律规则"和"指导原则"，前者不可商量，后者允许有据理力争
- 增加"正当例外申报"机制：AI 可以声明跳过某规则，但必须给出理由并记录

#### 7. 没有学习/反馈机制

Skills 是 **静态文档**。系统没有机制在执行过程中 **自我学习和改进**：

- 哪些任务分解方式经常导致审查不通过？
- 哪些类型的代码审查反馈反复出现？
- 哪些 brainstorming 问题对最终设计影响最大？

`verification-before-completion` 中提到的 "24 条失败记忆" 是作者 **手动策展** 的经验总结，不是系统自动收集的。

**改进建议：**
- 添加 `retrospective` skill：在每次开发周期结束时自动总结经验教训
- 支持 skill 的"动态补丁"：根据执行历史自动增加新的反模式警示

#### 8. 技能覆盖盲区

14 个 skills 覆盖了核心开发流程，但遗漏了很多常见场景：

| 缺失的场景 | 说明 |
|-----------|------|
| 数据库迁移 | 如何安全地做 schema 变更 |
| 部署/发布 | CI/CD 集成、蓝绿部署、回滚策略 |
| 文档编写 | API 文档、用户手册、架构文档 |
| 性能优化 | profiling、基准测试、瓶颈分析 |
| 安全审查 | 漏洞扫描、依赖审计、权限模型 |
| 依赖管理 | 升级策略、破坏性变更处理 |
| 重构策略 | 大规模重构的分步方法 |
| 多人协作 | 多个 AI 代理协同开发同一项目 |

**改进建议：** 优先补充部署、文档、安全审查相关 skill

#### 9. 用户体验的疲劳问题

Brainstorming skill 要求 **"一次一个问题"** 和 **"分段呈现设计，逐段审批"**。

对于经验丰富、已经很清楚自己想要什么的开发者，这个过程可能极其恼人：

```
AI: "你想用什么语言？"
用户: "Python"
AI: "你想用哪个框架？"
用户: "FastAPI"
AI: "你想怎么处理认证？"
用户: "JWT"
AI: "好的，让我呈现设计第 1 部分…这样对吗？"
用户: "对"
AI: "第 2 部分…对吗？"
用户: (崩溃) "我30分钟前就可以开始写代码了！！"
```

**改进建议：**
- 支持 "expert mode"：允许用户一次性提供完整需求，跳过逐步提问
- 检测用户的详细程度：如果用户已经给出了非常详细的需求，减少提问
- 允许用户说 "不需要讨论，直接按我说的做"

#### 10. 灾难恢复缺失

整个框架没有讨论 **子代理完全失控** 的场景：

- 子代理删除了重要文件怎么办？
- 子代理引入了安全漏洞但通过了审查怎么办？
- 控制器（主代理）自身上下文溢出导致忘记了计划怎么办？
- 连续多次审查循环不通过，陷入死循环怎么办？

**改进建议：**
- 增加 `disaster-recovery` skill：当子代理失控时如何回滚
- 设置审查循环上限（如 3 次不通过就暂停并请求人工介入）
- 关键操作前创建 git tag 作为安全回滚点
- 监控控制器上下文使用量，在接近上限时主动保存状态

#### 11. 审查者之间的潜在冲突

`subagent-driven-development` 中，规格审查和代码质量审查是 **串行的两个独立子代理**。但文档没有讨论它们可能冲突的情况：

```
场景: 
  规格审查者: "代码必须实现 retry 带指数退避" ✅
  代码质量审查者: "retryWithExponentialBackoff() 函数太复杂，
                   应该拆成更小的函数"

  → 实现者拆分函数以满足质量审查
  → 规格审查者重新审查: "函数签名变了，不符合原始计划的接口设计"
  → 循环…
```

**改进建议：**
- 允许规格审查和质量审查 **跨审查沟通**
- 在每一轮修复后同时触发两个审查，而不是串行

---

**最终评价：** Superpowers 在"如何让 AI 代理遵循软件工程最佳实践"方面做得非常出色，其反合理化防御和流程编码化的设计思路值得所有 AI 工具开发者学习。但它的主要软肋在于 **过度理想化** —— 假设每个项目都适合完整流程、每个代码库都测试健全、每个用户都有耐心配合苏格拉底式提问、token 成本不是问题。对于真实世界的混乱代码库和赶工的项目节奏，它需要更多灵活性和降级策略。
