# dev-skills-kit

`dev-skills-kit` 是一个为团队和个人开发者打造的跨平台 AI Agent 辅助配置与工作流分级适配系统。旨在于多辅助平台（如 Cursor, Antigravity, Codex, OpenCode 等）下，缓解 AI 行为差异、上下文占用过高以及团队开发规范难以统一落地的问题。

---

## 为什么开发这个工具？(立项背景与痛点)

在全面拥抱 AI 辅助开发的日常中，我们面临着以下三大痛点：

1. **各类 AI 平台能力分化**
   不同的 AI 辅助开发工具能力差异较大。例如，Cursor 拥有 Composer 模式，OpenCode 内置了多子代理（Subagent）调度机制，而诸如 Antigravity 等部分基础工具往往缺少此类自主调度支持。统一的提示词方案既难以充分利用高权限平台的优势，又可能在使用受限平台时产生预期外的行为或幻觉。
2. **"胖配置"导致的上下文堆积 (Context Decay)**
   常见的做法是将多种约定、架构规范组合在一个大型 `AGENTS.md` 或 `.cursorrules` 文件中。这种整合不仅会在每次对话启动时占用大量 Token，且随着会话行数的累积，AI 也容易"遗忘"文档底部的详细防御性指令。
3. **团队级开发规范难以统一对齐**
   在形成了较好的 Prompt 调度技巧、工作流（如 TDD 流程、Bug 排查 4 步流）或设计规范后，缺乏集中化维护机制会导致团队成员靠复制粘贴进行同步。随着时间推移，不同项目间的开发标准极易产生偏离。

---

## 本系统的主要特点

为了缓解上述问题，`dev-skills-kit` 从配置组织层面尝试提供了以下支持：

### 1. 模块化的关注点分离
将传统的超大提示词文件进行拆分。系统将不同语言栈的专业规范、通用架构模式和工作流方法论（Skills）拆分为独立的 `.md` 文件（如 `.agent/skills/xxx/SKILL.md`）。
AI 仅在评估当前具体任务有需要时，才会按需读取。这有助于节约上下文窗口，减少智能体在长对话中的遗忘现象。

### 2. 多平台环境适配
系统对底层规则进行了 Common 组件与 Platform 组件的拆分机制（位于 `.agent/builder/`）：
- 针对**内置能力较强的生态（Cursor, OpenCode, Codex）**：提供精简版的基础配置，让智能体更多利用其自身的原生调度和规划能力。
- 针对**偏基础的辅助生态（Antigravity）**：提供内嵌步骤限制的基础版配置，以提高执行时的稳定性。

### 3. 按需的平台配置接入
利用不同 IDE 平台的特定规则读取路径（如 `.cursor/rules/dev-skills-kit.mdc`、`.opencode/AGENTS.md` 等），生成引导文件。这可以让各家 AI 助手进入项目后，能够自动去读取匹配该平台的配置文件。

### 4. 统一的分发与更新
通过提供统一的安装脚本外加跨平台相对路径引用，你可以将一套统一的 AI 开发指南注入到现存的旧项目中：
```bash
~/dev-skills-kit/install.sh /path/to/your-project
```
当配置发生更新时，在中央仓库拉取最新代码后再执行此脚本，即可完成所负责项目的同步。

## 设计理念

### 按需加载，而非全量注入

传统方案把所有规则塞进一个大文件，每次会话都全量加载。`dev-skills-kit` 的做法是：

```
AGENTS.md（轻量骨架，始终加载）
  └─ 按需读取 → .agent/skills/xxx/SKILL.md（仅在需要时加载）
```

AGENTS.md 只定义调度逻辑和场景分级规则，具体的 Skill 指导（TDD 流程、调试方法论、编码规范等）以独立文件存在，AI 根据当前任务**按需读取**。这样既不浪费 token，又避免了长会话中的规则遗忘。

### 场景分级，拒绝一刀切

不同复杂度的任务不应走同一套流程：

| 复杂度 | 流程 | 示例 |
|--------|------|------|
| **trivial** | 直接做 → 验证 | 改变量名、修 typo |
| **small** | 调试 + 验证 + 灵活 TDD | 修 bug、添加字段 |
| **medium** | 设计 → 计划 → 分批执行 → 审查 | 新增 API、业务模块 |
| **large** | 完整方法论 + 规范归档 | 系统重构、数据库迁移 |

改个变量名不需要 6 步流程，重写认证系统也不该跳过设计评审。

### 一套源码，多平台适配

各 AI 平台的能力和配置方式差异很大：

```
.agent/builder/
├── common/          ← 共享的核心规则（场景分级、Skills 注册表、缺陷补丁）
└── platforms/
    ├── antigravity/  ← 无子代理调度，用 executing-plans 替代
    ├── cursor/       ← 有 Composer，精简配置
    ├── codex/        ← 有原生代理，精简配置
    └── opencode/     ← 有 subagent，启用原生调度
```

`build.sh` 将 common + platform 拼装为各平台专属的 AGENTS 配置文件，通过各平台的拦截规则（`.cursor/rules/`、`.codex/agent.rules` 等）自动引导 AI 读取对应版本。

### 零 Hooks 依赖

项目的所有功能（经验沉淀、状态持久化、规则加载）都**不依赖** Claude Code Hooks 等平台特有机制。经验沉淀通过 `auto-learning` skill 实现——任务完成时自动提取经验到 `docs/learnings/` 目录，新任务开始时按文件名检索相关经验。纯文件驱动，所有平台通用。

### 防遗忘设计

AI 在长会话中会逐渐"遗忘"规则。系统通过三层机制对抗：

1. **planning-with-files**：3 个持久化文件（`task_plan.md` / `findings.md` / `progress.md`），即使上下文丢失也能恢复状态
2. **定期刷新**：每完成一个阶段，重新读取计划文件
3. **`/go` 指令**：用户随时可触发规则重载，将 AI 拉回正轨
